C4 Level 1 — Thunderbird for iOS talks to mail providers directly; Thunderbird the organization operates no mail relay in this path.

🔑 Key architectural fact The app is a direct mail client. There is no proprietary sync server between the device and the user's mailbox — the engine speaks IMAP/SMTP/JMAP to the provider itself. This shapes every level below: networking, security, and offline behavior all live on-device.

Actors

The single human actor. Brings one or more existing email accounts (the app does not provision mailboxes). Goals: add accounts quickly, see new mail, read messages with proper rendering and attachments, and manage folders. Interacts only through the SwiftUI app on iOS 18+.

External systems

The classic pairing: IMAP4rev1 for reading/synchronizing mailboxes and SMTP for submission. The engine speaks both over TLS or STARTTLS via SwiftNIO. Used for any provider without JMAP, and for OAuth2 providers such as Google, Microsoft, and Yahoo.

📄 Code Core/Sources/IMAP/IMAPClient.swift · Core/Sources/SMTP/SMTPClient.swift

The modern JSON-over-HTTP mail protocol (RFC 8620). A single batched request can fetch mailboxes, query messages, and resolve threads — no per-mailbox SELECT round-trips. Detected for Fastmail and used by Thundermail. Preferred over IMAP when available for an account.

📄 Code Core/Sources/JMAP/JMAPClient.swift

Token-based sign-in instead of passwords. The app ships pre-registered client configurations for AOL, Fastmail, Google, Microsoft, Thundermail (Keycloak), and Yahoo. The provider is chosen by matching the email domain's MX records, then an ASWebAuthenticationSession performs the browser-based flow.

📄 Code Thunderbird/Thunderbird/Account/OAuth2.swift · Core/Sources/Autoconfiguration/OAuth2.swift

So users type only an email address. Sources are tried in trust order: the provider's own .well-known/autoconfig, the Mozilla ISPDB (autoconfig.thunderbird.net), and — as a fallback — DNS MX lookup to derive the real provider domain. Results are decoded from Thunderbird's autoconfig XML into a typed ClientConfig.

📄 Code Core/Sources/Autoconfiguration/URLSession.swift · Core/Sources/Autoconfiguration/XMLToJSONParser.swift

Per-distribution JSON hosted on GitHub Pages (thunderbird.github.io/thunderbird-ios/feature_flag_config_{debug,beta,appstore}.json). Lets features be toggled remotely without a release, gated by a user opt-in for remote flags.

What's in scope

✓ In the system

  • Account onboarding & autoconfiguration
  • IMAP/SMTP and JMAP connectivity
  • Mailbox & message listing, reading, HTML rendering, attachments
  • OAuth2 and password authentication
  • On-device credential storage
  • 69-language localization

○ Outside / emerging

  • No Thunderbird-operated sync backend
  • Compose/send UI is present in the engine; app compose is still landing
  • OAuth2 token exchange in-app is partially stubbed (see Level 4)
  • Push/IDLE wired in the engine; not yet surfaced as background refresh

Why these choices

Privacy and trust. Mail flows only between the device and the user's own provider; Thunderbird never sees message content. The cost is that synchronization, threading, and connection management must all be solved client-side — which is exactly what the Core engine exists to do.

IMAP is universal; JMAP is better. Supporting both means the app works with every provider today while delivering a faster, batch-friendly experience where JMAP exists. The Account module hides the difference behind unified Server, Mailbox, and message models so the UI is protocol-agnostic.
