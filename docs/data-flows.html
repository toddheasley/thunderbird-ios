Runtime flows

The levels describe structure; these describe behavior. Each flow traces a single user-visible action through the layers — app view → Account facade → protocol module → external system — so you can see where responsibilities hand off.

  1. Launch with no accounts — ContentView sees hasAuthorization == false and shows WelcomeScreen, which presents ManualAccount as a sheet.
  2. Enter email — AccountInformation captures the address into the shared LoginDetails.
  3. Discover settings — AutoconfigView calls Account.autoconfig(_:) (see the Autoconfiguration flow). On success it pre-fills servers; otherwise the user picks a protocol in EmailAccountTypeSelection and fills ManualServerSetup.
  4. Authenticate — AuthorizationView collects a password or runs OAuth2; the credential is assigned to Server.authorization, which writes it to the keychain.
  5. Persist — accounts.set(account) stores account metadata to disk via FileManager.
  6. Enter the app — hasAuthorization flips true; ContentView swaps to EmailListView.

URLSession.autoconfig(emailAddress:sources:queryMX:) tries sources in trust order and returns the first match:

  1. Provider .well-known — fetch https://<provider>/.well-known/autoconfig/mail/config-v1.1.xml.
  2. Mozilla ISPDB — fall back to autoconfig.thunderbird.net, the same database Thunderbird desktop uses.
  3. MX fallback — if still unresolved and queryMX is on, a DNS MX lookup (swift-async-dns-resolver) finds the real mail host; DomainParser reduces it to a registrable domain via the public-suffix list, and discovery retries.
  4. Decode — XMLToJSONParser converts the autoconfig XML to JSON, decoded into ClientConfig (EmailProvider + servers + OAuth2 params).

Account.autoconfig(_:isJMAPAvailable:) wraps this, adds a Fastmail-JMAP special case, and returns a populated Account.

🛠Try it from the terminal The AutoconfigurationCLI executable runs this exact pipeline for any address — handy for debugging a provider without launching the app.
  1. Match provider — OAuth2.request(_ emailAddress:) does an MX lookup and matches the host against built-in configs for AOL, Fastmail, Google, Microsoft, Thundermail, Yahoo.
  2. Configure button — OAuthButton.configure() resolves the OAuth2.Request (authorization URL, scope, redirect).
  3. Browser auth — authenticate() opens an ASWebAuthenticationSession against the provider's auth URL with callback scheme <bundle-scheme>://oauth2redirect.
  4. Token — the returned code is exchanged for a bearer token, set on the binding and stored via Server.authorization = .oauth(...) into the keychain.
🚧Status Provider matching and the browser session are wired; the final token-exchange step is partially stubbed in-app (see Account/OAuth2.swift).

Account.emailProtocol chooses the path — JMAP when the account supports it, otherwise IMAP. The UI calls the same facade either way.

IMAP path

  1. imapClient (cached by UUID) connect() + login()
  2. list() mailboxes, select(mailbox:) the target
  3. fetch(uid:attributes:) streams messages via NIO handlers
  4. Bodies decoded through MIME into displayable content

JMAP path

  1. session(server:) once
  2. mailboxes() → emails(in:) → emails(_:), batched
  3. thread(for:) resolves conversations
  4. Server returns parsed structure — minimal client-side MIME

Both converge on the unified Account.Mailbox/message shape the views consume.

  1. Build — an SMTP.Email (sender, recipients, subject, body, date) is constructed from the compose data.
  2. Bridge — the account's outgoing Server converts to an SMTP.Server (credentials pulled from the keychain).
  3. Submit — SMTPClient.send(email:) opens a NIO connection; SendHandler walks the state machine (EHLO → STARTTLS → AUTH → MAIL FROM → RCPT TO ×N → DATA → QUIT). See Level 4 · SMTP.
  4. Result — success closes the connection; failures surface as typed SMTPError cases for the UI.
🧭Engine vs app The send pipeline is complete in Core/SMTP; the in-app compose surface that drives it is still landing (the list view's compose FAB is currently disabled).
