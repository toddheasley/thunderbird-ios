Components

Open up each container and name the components inside — the modules, view groups, and the principal types that carry the responsibilities. This is where the architecture becomes concrete.

Core engine components

Seven library modules form a clean dependency layering: two leaf modules (EmailAddress, MIME), three protocol clients that build on them (IMAP, SMTP, JMAP), a discovery module (Autoconfiguration), and the Account facade that unifies them all.

Leaf · EmailAddress, MIME
No internal dependencies — shared value types
Protocol clients · IMAP, SMTP, JMAP
depend on MIME + EmailAddress (+ SwiftNIO for IMAP/SMTP)
Discovery · Autoconfiguration
depends on swift-async-dns-resolver
Facade · Account
composes all of the above into unified Account / Server / Mailbox + client pooling

Responsibility: one shared address model so IMAP, JMAP, and SMTP don't each reinvent it.

  • EmailAddress — struct with an optional display label and the address; Codable, Hashable, Identifiable, and ExpressibleByStringLiteral so "Name <a@b.com>" parses directly.
  • EmailAddress.Group — a named group of addresses (RFC group syntax).
  • EmailAddressProtocol — abstraction for polymorphic address collections.

Responsibility: parse and build MIME message parts — multipart boundaries, content types, transfer encodings, attachments.

TypeRole
PartA single MIME part (headers + encoded body); RawRepresentable between Data and structured form
BodyMultipart body; recursively decodes child parts on the boundary
ContentTypeMedia type enum (text/image/…); carries boundary & charset
ContentTransferEncodingbase64, quoted-printable, …
ContentDispositioninline vs. attachment + filename
MIMEErrorparsing/encoding failure cases

Headers are matched case-insensitively/fuzzily for robustness against real-world mail.

Responsibility: a full IMAP4rev1 client. Connection lifecycle, login, capability negotiation, mailbox listing/selection, message fetch (by sequence number or UID), flag updates, and IDLE push.

TypeRole
IMAPClientOwns the NIO channel; runs tagged commands; exposes connect, login, list, select, fetch, idle, noop
ServerHost, port, credentials, ConnectionSecurity (TLS/STARTTLS/plain)
Message / EnvelopeParsed message: flags, UID, envelope (from/to/subject/date), body
MailboxName, path, attributes, status
Flag\Seen, \Flagged, \Draft, \Deleted, …
IMAPCommandProtocol for each command (Login, Capability, List, Select, Fetch, UIDFetch, Idle, Store, Create, Delete, Rename, Status, Noop)
IMAPErrornotConnected, timedOut, commandFailed, unexpectedResponse, serverDisconnected, …

Built on swift-nio-imap for the protocol codec. Fetch responses stream through NIO handlers and assemble incrementally rather than buffering whole messages. Detail Level 4 · IMAPClient.

Responsibility: submit a message. A NIO handler pipeline drives the SMTP conversation as a state machine.

  • SMTPClient — send(email:), iterating one RCPT TO per recipient.
  • Email — sender, recipients, subject, body, date.
  • SendHandler — the state machine: greeting → EHLO → STARTTLS → AUTH LOGIN → MAIL FROM → RCPT → DATA → QUIT. Supported by RequestEncoder, ResponseHandler, MessageHandler, LoggingHandler.
  • SMTPError — requiredTLSNotConfigured, emailRecipientNotFound, remoteConnectionClosed, responseNotDecoded.

Detail Level 4 · SMTP send state machine.

Responsibility: a JMAP (RFC 8620) client. Stateless JSON method calls instead of stateful IMAP sessions.

TypeRole
JMAPClientsession(server:), mailboxes(), emails(in:), emails(_:), thread(for:)
SessionFrom /.well-known/jmap — accounts, capabilities, API URL
EmailblobID, threadID, addresses, keywords, size, dates + GetMethod/QueryMethod
Mailbox / ThreadRoles, rights, counts / thread email IDs
AuthorizationBearer token or OAuth2
Filter / Email.ConditionQuery filters (inMailbox, hasKeyword, before/after, text)
JMAPErrormethod / request / set / underlying URL errors

Methods serialize to the JMAP ["Email/get", {…}, "callId"] tuple form and POST in a batch via a URLSession extension. Detail Level 4 · JMAP batch requests.

Responsibility: turn an email address into server settings, trying trusted sources in order.

  • URLSession.autoconfig(emailAddress:sources:queryMX:) — the entry point.
  • Source — provider .well-known → Mozilla ISPDB → MX fallback.
  • EmailProvider / EmailProvider.Server — discovered config (host, port, socket type, auth methods).
  • ClientConfig — aggregate result (provider + OAuth2 + webmail).
  • XMLToJSONParser — converts Thunderbird autoconfig XML to decodable JSON.
  • DomainParser / SuffixListParser — public-suffix-aware domain extraction.
  • SRVResolver / DNS MX queries via swift-async-dns-resolver; OAuth2 request params.

Also home to the AutoconfigurationCLI executable for testing discovery from the command line. Detail Flows · Autoconfiguration.

Responsibility: the single API the app uses. Models accounts & servers protocol-agnostically, stores credentials in the keychain, pools live clients, and runs autoconfiguration.

TypeRole
AccountIdentities + servers; emailProtocol picks JMAP when available else IMAP; incomingServer/outgoingServer; autoconfig(_:)
ServerUnified server for any protocol; bridges to IMAP.Server, SMTP.Server, JMAP.Server; keychain-backed authorization
ServerProtocolimap · jmap · smtp
MailboxUnified mailbox with Role (inbox/drafts/sent/trash/junk/archive) and Rights; built from IMAP or JMAP mailboxes
AccountManager / Accounts@Observable store — persist to disk via FileManager, add/update/delete
Authorization / Tokenbasic(user,password) · oauth(user,token) · none
MailboxManagerPer-account mailbox state

Clients are cached per account UUID (imapClient/jmapClient) and reconnect/login lazily. Detail Level 4 · Account & keychain.

App components

The app target groups its 32 SwiftUI views into feature folders. Navigation is driven entirely by SwiftUI state — there is no coordinator object.

  • App (@main) — holds @State accounts: Accounts and @State featureFlags: FeatureFlags(distribution:), injects both into the environment, and overlays a global FeatureNotImplementedView driven by AlertManager.shared.
  • ContentView — computes hasAuthorization from accounts.allAccounts (both incoming & outgoing servers authorized). True → EmailListView; false → a NavigationStack with WelcomeScreen presenting ManualAccount as a sheet.

A string-path NavigationStack rooted at ManualAccount, threading a shared @Observable LoginDetails (in-progress account, entered email, chosen protocol):

  1. AccountInformation — email entry; AutoconfigView attempts discovery; on success shows AuthorizationView, else offers manual setup.
  2. EmailAccountTypeSelection — JMAP vs IMAP toggle (custom FullToggleStyle).
  3. ManualServerSetup — incoming/outgoing host, port, security, auth; finalizes via accounts.set(account).

Authentication UI: AuthorizationView switches on AuthenticationType — password (SecureField) or OAuth2 (OAuthButton → ASWebAuthenticationSession). Provider configs live in Account/OAuth2.swift. Detail Flows · OAuth2.

AccountList — drawer navigation

  • DrawerView — slide-in sidebar with dimmed overlay; links to GeneralSettingsView.
  • DrawerContent — static Inbox/Sent rows + one AccountFolderDisclosureView per account.
  • MailboxDropdownRowView / AccountFolderDisclosureView — DisclosureGroup mailbox trees.

EmailDisplay — list & reading

  • EmailListView — selectable List of cells with edit-mode multi-select, sort menu, drawer toggle, and a (disabled) compose FAB.
  • EmailCellView — row preview: sender, subject, body snippet, smart date, and pin/unread/attachment/thread indicators.
  • ReadEmailView — the largest view (625 lines): SenderView header, a WKWebView for HTML body, AttachmentBlockView, plus reply/forward/archive/flag/move toolbars.
  • TempEmail (@Model, SwiftData) — the in-app email model (addresses, subject, HTML body, date, flags, attachments) with sampleData for previews. Real fetches from Core are being wired in over this shape.
  • AlertManager (@MainActor @Observable singleton) — global “not implemented”/error alerts.
  • FeatureFlags (@MainActor @Observable) — local featureSettings persisted to UserDefaults, optionally merged with remote JSON; Distribution.current (debug/beta/appstore) selects the config URL. Flag enum keys (e.g. fullDate) are queried with flagForKey(_:). FeatureFlagDebugView exposes toggles in debug builds.
  • SmartDateFormatter — relative/absolute date formatting gated by the fullDate flag.

Bolt components

  • Color — semantic palette with light/dark variants: neutral(.base/.lower/.border/…), intents primary/secondary/success/warning/critical (soft/default/hover/pressed), text(.base/.muted/…), and an accent set (teal/blue/purple/orange/pink/ink).
  • Font.Style — scale from titleLarge (31pt) to caption (11pt), each with size/lineSpacing/kerning/weight; applied via .textStyle(_:isDynamic:) with Dynamic Type support.
  • Foundation — hex⇄Int color helpers and RGB component extraction.
  • Editor — reserved module for the rich-text compose editor (re-exported through BoltUI).
